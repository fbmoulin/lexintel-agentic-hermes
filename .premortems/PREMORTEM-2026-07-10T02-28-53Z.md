### Finding 1: Systemic index outage completes every case as a "warning", never an alarm

**Category:** 7 (Invisible invariants) + 5 (Coincidental correctness), Qdrant addendum (non-atomic upsert / orphaned vectors)
**Severity:** medium
**Confidence:** confirmed
**Location:** `app/agents/indexing_agent.py:44-77`
**Mitigation verified absent:** the `except Exception` at `indexing_agent.py:46` catches *every* upsert error — transient and systemic alike — and returns `status="warning"` with `requires_human_review=True`, then the orchestrator continues (it halts only on `blocked`, `orchestrator.py:281-287`). There is no distinction between "one chunk failed" and "Qdrant is down", no counter/metric/alert, and nothing that reconciles "run completed" against "chunks actually indexed". `grep -n "recreate_collection\|score_threshold\|reconcile\|metric\|alert" app/` → 0 matches in the indexing path.

#### Failure narrative
The team flips `LEX_KRATOS_ENABLE_QDRANT=true` in a real deployment (the whole point of PR #17). Qdrant then goes unreachable for an hour (restart, network, disk-full). Every `run_full_mock` call still runs Intake→Security→…→FIRAC→Validation and returns `status="warning"` — the *same* status a single low-quality extraction produces — so the dashboards look normal and nobody pages. Meanwhile the invariant a future consumer assumes — **"a pipeline run that completed ⇒ its chunks are in the vector index"** — is silently false for every case processed during the outage. The already-planned `HybridRetrievalAgent` (`registry.py:76-84`, status `planned`) is built later on exactly that assumption; it retrieves against an index missing an hour of cases and returns confidently wrong precedent context, with no error anywhere. The best-effort decision (M1) is correct for *this* case's analysis, but it converted a loud `failed` into an indistinguishable `warning`, erasing the only signal that would have surfaced a systemic backlog.

**Hardening:**
- Tag the degraded result distinctly (e.g. `output["index_status"]="upsert_failed"`) so a systemic run of index failures is queryable/alertable separately from content warnings.
- When the real Qdrant backend is enabled, emit a metric/log-once on upsert failure and add a reconciliation/backfill job before any retrieval agent depends on index completeness.
- Document the now-broken invariant at `indexing_agent.py` and at the `HybridRetrievalAgent` registry entry so the future author sees it.
